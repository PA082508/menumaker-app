## 0. Заказ в одну фразу

7 учителей-родителей заполняют пакеты; в Inbox прилетит порядка **70–80 форм**. Плоский
список из 80 карточек пугает и не читается. Свернуть его в **дерево семья → ребёнок**, где
три подписных формы помечены, а «подписал → одно действие подшивает весь пакет ребёнка/семьи
в БД». Ручной осмотр остаётся, но директор работает **блоками**, а не строками.

Это НЕ авто-файл из renewal-контура: подшивка здесь — **ручное пакетное Approve**, действие
директора, статус `approved`. Claim-мост цел.

## 1. Модель данных — два слоя

Формы делятся не «по детям», а по **уровню**:

| Слой | Кто заполняет | Формы |
|---|---|---|
| **family** (одна на домохозяйство) | родитель раз | `iea`, `usda_waiver` (после переделки, §3), информационные (`parents_book_ack`) |
| **child** (своя у каждого ребёнка) | по разу на ребёнка | `cacfp_enrollment`, `parent_consent`, `dcy_01234`, медицинские (`dcy_01236/01217/01305`, `special_diet`) |

**Решение:** ввести в реестр (`public/enroll-registry.json`) атрибут **`scope: "family" | "child"`**
у каждой формы. Информационные = `family`. Это метаданные — дёшево, живую БД не трогает.

Заметка: `scope` — про **сколько сабмитов приходит**, не про то, какой слот форма закрывает.
`usda_waiver` закрывает семейный income-слот, но до §3 приходит по разу на ребёнка. После §3 —
`scope: family`.

## 2. Якорь семьи — что объявляет состав

Общего `packet_id`/`session_id`/`family_id` на `enrollment_submissions` **нет** (проверено:
каждая форма — независимая строка со своим `idempotency_key`). Prefill-токен связывает формы,
но только для renewal **существующего** ребёнка; новых детей — сегодняшний случай — он не
покрывает. Значит семью выводим из того, что её **объявляет**, по убыванию надёжности:

1. **`form_data.children[]`** из income-формы (IEA или переделанного waiver, §3) — **авторитетно**.
   `validateIea` уже читает `children[]` — переиспользуем.
2. **`parent_name`** (waiver, согласия, book-ack) — мягкая группировка, когда income-формы ещё нет.
   Присутствует не на всех типах (нет на `cacfp_enrollment`/`iea`-строке), потому только запасной.
3. **`child_name`** — лист дерева, если и родитель не сматчился.

**Namesake-риск (однофамильцы, опечатки) допустим — это группировка для ОТОБРАЖЕНИЯ.**
Мисматч директор видит глазами и чинит. Это принципиально иначе, чем авто-файл, где ошибка
уходит в БД молча. Матчинг к реальному `child_id` при подшивке — по-прежнему через
`matchRoster` (точное имя + fuzzy c подтверждением по дате рождения),
[enrollmentApprove.ts:279](../../src/lib/enrollmentApprove.ts#L279).

## 3. USDA waiver → семейный (v3)

**Факт:** `usda_waiver` — **наше изобретение**, не официальная USDA-форма (реестр:
`requiringOrg: "Internal"`). Текущая v2 — **на одного ребёнка**: поля `Child's Name`, `Date`,
`Parent Name (print)`, `Parent Signature`, галочка «I DO NOT qualify…». Детей не перечисляет.

**Назначение формы** — не «этот ребёнок Paid», а **сходимость охвата опросом**: IEA закрывает
одних детей, waiver — остальных, вместе = 100% детей спрошено.

**Решение:** переделать waiver в **семейный, с перечислением детей** — `children[]`, в **той же
структуре, что у IEA**. Формат — **фиксированные строки, до 4, зеркалом IEA** (`kid1..kid4`, имя +
дата рождения), **репитер не строим**; имя+DOB обязательны для первого ребёнка. НЕ «через запятую».

Почему структурные строки, а не запятая:
- Якорь обязан привязываться к реальным строкам ростера (`child_id`). Блоб имён пришлось бы
  снова разбирать — тот же namesake/опечатка-ад; по дате рождения не подтвердить.
- Одинаковая с IEA структура `children[]` ⇒ **код якоря и матчинга — одна ветка, а не две**.

**Лимит 4 и семья >4 детей (названо, не подразумевается).** Форма несёт ровно 4 строки, как
IEA. Домохозяйство с 5+ детьми **подаёт waiver второй раз** на остальных; **группировка по
`parent_name`** в дереве инбокса склеивает обе подачи в одну семью (§2, запасной якорь). В самой
форме это сказано паренту строкой «More than four children? …submit it again». То же ограничение
действует и для IEA (тоже `slice(0,4)`).

Реализация DARK: `USDA_Waiver_v3.html` собран на ветке `feat/usda-waiver-v3-family` (Pages-репо),
`collect()` шлёт `children:[{name,dob}].filter(name)` + `child_name`=первый ребёнок (совместимость
с плоским инбоксом до релиза дерева). Registry `current` остаётся v2 — flip после сверки → smoke.

**Побочная ценность — сигнал покрытия.** «Чтобы в общем подсчёте сходилось» становится метрикой:

```
(дети во всех IEA) ∪ (дети во всех waiver)   vs   активный ростер
→ кто ещё не опрошен
```

Ложится на инвариант полноты ростера ([[menumaker-claim-bridge-invariant]]). Кандидат в плашку
инбокса/дашборда: «N детей без income-ответа». (Реализация плашки — отдельным заходом, не блокер.)

**Обоснование для коммерческой версии (ОБЯЗАТЕЛЬНО в инструкции продукта).**
Форма — наша, не официальная USDA, поэтому покупатель СПРОСИТ «зачем она». Обоснование:
USDA не предписывает *эту* форму, но CACFP требует, чтобы спонсор **опросил на income-eligibility
каждого зачисленного ребёнка**. Waiver фиксирует **подписанный отказ**, чтобы: (а) у каждого
ребёнка было определение на руках; (б) спонсор мог показать на проверке штата **полное покрытие
опросом** (нет «немых» дыр в ростере); (в) over-income домохозяйства были задокументированы как
**Paid**, а не молча отсутствовали. То есть форма — инструмент **аудит-готовности и полноты**, а
не бюрократия. Этот абзац завести в инструкцию коммерческого варианта (Doc Hub / onboarding).

**Итоговая модель якоря — симметричная:**
`iea` → `scope: family`, `children[]` → якорь · `usda_waiver` (v3) → `scope: family`, `children[]`
→ якорь **тоже** · `parent_name` → запасной · `child_name` → лист.

Эквивалентность слота дохода уже держит `satisfies: ['iea']` на waiver — у каждого перечисленного
ребёнка income-слот закрывается либо общей IEA, либо семейным waiver.

## 4. Inbox UI — дерево семья→ребёнок

Сегодня [EnrollmentInboxPage.tsx:144](../../src/pages/enrollment/EnrollmentInboxPage.tsx#L144) —
плоский список, одна карточка на сабмит, сортировка `created_at desc`, группировки нет; имя для
показа/поиска берётся из `form_data.child_name`. Меняем на **дерево**:

```
▸ Семья  (якорь = children[] из income-формы, иначе parent_name)
    Семейные формы: IEA / USDA waiver, Parents Book — подшиваются ОДИН раз
    ▸ Ребёнок A → его формы   (3 подписных помечены ✍️)
    ▸ Ребёнок B → его формы
    [ Подписать 3 и подшить всё ]
```

- **Рама берётся готовой** (headless tree-table с sub-rows, напр. TanStack Table) — раскрытие/
  вложенность/группировка из коробки, не пишем руками. **Мотор (якорь, подписи, подшивка) — наш.**
- Раскрытие семьи/ребёнка по клику (и по наведению).
- Заголовок блока: имя + счётчик форм + индикатор «3 ждут подписи».
- **Три подписных формы (список A, §6): `dcy_01234`, `iea`, `start_form`** — явная пометка
  «✍️ нужна ваша подпись». Остальные — «готово к подшивке».
- `received`/уже подшитые внутри блока показываются как факт, не как задача (не входят в
  красную цифру — инвариант renewal-контура).

## 5. Оркестрация «подшить всё»

Формы **уже в БД** (`status='pending'`). «Подшить всё» = **пакетное ручное Approve поверх
существующих `approve*`** ([enrollmentApprove.ts](../../src/lib/enrollmentApprove.ts)), в один клик
на блок вместо N кликов. Порядок **жёсткий** (зависимость по `child_id`):

1. **Детские `cacfp_enrollment`** → `approveCacfpInsert` создаёт строки ростера → получаем `child_id`
   каждого ребёнка.
2. **Семейная income-форма** (IEA `approveIea` / waiver) → файлится **один раз на всех** сматченных
   детей. `approveIea` уже пишет на **каждого** сматченного ребёнка — «одна форма → на всех» заложено.
   ⚠️ Определение F/R/P (`roster.frp` + `income_eligibility`) — **всегда рукой директора**, под
   claim-мостом до 1 октября; авто-файлить его нельзя ни при каких флагах (§6).
3. **Индивидуальные согласия/документы** → `approveDocument` (привязка `child_id`, мёрж подписи
   директора для подписных, никогда не перезапись существующего слота).

Три подписных формы требуют подписи директора **до** шага подшивки — иначе блок не проводится
(`approveDocument` мёржит контрподпись в `signatures`).

## 6. Инварианты и что развести до включения

- **Claim-мост (до 01.10):** определение F/R/P никогда не авто-филится; при пакетном Approve оно
  идёт как **явное действие директора**, не как побочный эффект. IEA как «документ получен» — можно;
  определение — руками.
- **Namesake = только отображение.** Группировка по имени НЕ пишет в БД; привязка к `child_id` —
  через `matchRoster` + подтверждение директора.
- **Два несогласованных списка контрподписи — свести в ОДИН источник до пакетного Approve:**
  - **Список A** — `COUNTERSIGN_SLOT` ([signatureSamples.ts:152](../../src/lib/signatureSamples.ts#L152)):
    `dcy_01234`, `iea`, `start_form` — **измерен по живым сабмитам** (куда физически лечь подписи).
  - **Список B** — `renewal_countersign_types()`
    ([20260717_renewal_wave1.sql:143](../../supabase/migrations/20260717_renewal_wave1.sql#L143)):
    `transition_into_program`, `dcy_01234`, `child_release_authorization` — **декларативный**.
  - Пересечение — только `dcy_01234`. **Истина для «моя подпись уходит в БД» = список A** (решение
    Николая 2026-07-17). Судьбу `transition_into_program` / `child_release_authorization` решить
    отдельно (дать реальный слот подписи ИЛИ снять `requires_countersign`); `start_form` — в списке A,
    значит держать от авто-подшивки, у него есть слот `admin_sig`.
- **IEA сама многодетная** (`form_data.children[]`): в дереве показывается в блоке **каждого**
  перечисленного ребёнка, но **файлится один раз** (идемпотентно по сабмиту).

## 8. Открытые вопросы

1. **Плашка покрытия опросом** (§3) — MVP на понедельник или отдельный заход? (склоняемся: отдельный)
2. **Смешанная семья** (часть детей IEA, часть waiver) — на практике домохозяйство выбирает один
   путь; считаем один income-документ на семью. Подтвердить, что смешанного не бывает.
3. **Waiver v3 форма** — какие именно поля в строке ребёнка (имя обязателен; дата рождения — сильно
   желательна для fuzzy-матча; нужна ли классная комната?).
4. **TanStack Table vs ручной аккордеон** — финально при реализации, по весу бандла.
5. **Инструкция коммерческого варианта** — вписать обоснование waiver (§3) в onboarding/Doc Hub;
   владелец подтверждает формулировку. Действие, не блокер релиза.
