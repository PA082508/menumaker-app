# Enrollment Approval Loop — Specification
2026-07-03 (v2, утверждена Николаем) · Архитектор + Николай · для CC
Цель: избавить директоров от рутинного ввода сведений о детях. Родители заполняют пакет → система валидирует → директор проверяет чек-лист и одним действием отправляет в БД. Плюс инструменты оперативного управления пакетом и данными.

## 0. Архитектурное решение (подтверждено Николаем)
Консолидация на одном Supabase-проекте **menumaker (trrmyqfpxntmgxnqkikp)**:
- Таблица `menumaker.enrollment_submissions` (перенос схемы из childcare-core: id, child_id nullable, center_id, submission_type, form_data jsonb, signatures jsonb, signature_date, status, created_at) + новые поля: `validation jsonb`, `reviewed_by`, `reviewed_at`, `reject_reason`, `source text ('online'|'paper_entry')`, `paper_signed_at`, `paper_signed_by`.
- Формы пакета на GitHub Pages переключаются на menumaker endpoint (файлы выходят новыми версиями — кеш Pages). Публичный доступ форм — через SECURITY DEFINER RPC (submit_enrollment_form), НЕ прямой anon-insert в таблицу; урок safepass_sign учтён.
- childcare-core остаётся read-only архивом исторических сабмишенов до конца миграции; существующие данные перенести одноразовым скриптом.
- Канонические roster/families — menumaker.

## Фаза 1 — Director's Inbox + валидация (ядро)
**Экран `/enrollment-inbox`** (роли director/office_manager, по своему центру):
- Список pending-заявок: ребёнок (или «новый»), тип формы, дата, источник, бейдж валидации.
- **Валидация-агент** (авто при поступлении + пересчёт при правке):
  - Реестр правил per form type (рядом с childFieldRegistry, файл `enrollmentValidationRules.ts`):
    - CACFP Enrollment: имя, DOB, ≥1 день с часами и приёмами, дата подписи, телефон, адрес.
    - IEA: Part 1 обязателен; логика ветвления — валидный 7-значный case (Part 2) ⇒ Part 3 не требуется; Part 3 заполнена ⇒ SSN last-4 обязателен, если не отмечен «no SSN»; частоты дохода валидны.
    - DCY 01234: по его реестру полей.
  - Форматные проверки: даты, телефон, 7 цифр case, zip.
  - **Подпись НЕ входит в обязательные онлайн-поля** — действует бумажный оборот: формы печатаются и подписываются на бумаге. Вместо этого при Approve директор отмечает чекбокс **«Paper signed & filed»** (пишется paper_signed_at/by). Так до отмены бумажного оборота; конфиг-флаг org-уровня `paper_flow_required` — чтобы потом выключить одним переключателем.
  - Результат в `validation jsonb`: `{errors:[], warnings:[], missing:[]}` → бейдж: 🟢 готово / 🟡 предупреждения / 🔴 ошибки (что именно не заполнено — списком, человекочитаемо).
- **Diff-view**: слева данные формы, справа текущие roster/families (или «новый ребёнок»); маппинг полей через childFieldRegistry; директор может исправить значение прямо в diff (опечатки родителей) — правки логируются.
- **Approve = «Отправить в БД»** (последнее действие, одна кнопка): upsert в roster + families, статус approved, reviewed_by/at. Approve заблокирован при 🔴 ошибках (🟡 — разрешён с подтверждением). **Ничего не пишется в roster до Approve.**
- Reject с причиной (текст пригодится для фазы 3).


## Фаза 1.5 — Фото-приём бумажных форм (второй вход в тот же Inbox)
Для семей, заполняющих формы ручкой на бумаге (бланк с ресепшн). Убирает ручную перепечатку офисом.
- Офис фотографирует заполненную подписанную форму (телефон/сканер) → загрузка в приложении → OCR-агент извлекает поля (переиспользовать конвейер receipt-OCR + detectAndCrop) → заявка в тот же Inbox: `source='paper_entry'`, скан приложен, извлечённые поля в form_data.
- Та же валидация и тот же diff-view; уверенность OCR по полю < порога → поле подсвечено «проверьте» в diff.
- Каналы онлайн/фото параллельны и равноправны; после признания электронной подписи (org-флаг `paper_flow_required=false`) фото-приём остаётся как второй вариант для «бумажных» семей.

## Дополнения к Фазе 1 (утверждены)
- **Детектор дубликатов при Approve**: fuzzy-match новой заявки против roster (имя NFD/транслит, DOB, телефон семьи) → «Похоже на существующего ребёнка X — обновить его или создать нового?». Обязательный шаг для заявок «новый ребёнок». (Урок: Kendzierski Colton / Colton James.)
- **Undo вместо подтверждений**: Approve/Reject исполняются сразу + toast «Approved · Undo» 10 сек (undo возвращает pending и откатывает upsert); блокирующий confirm — только для действий без отката.

## Фаза 2 — Чек-лист пакета (аналитика документов)
- Матрица «ребёнок × требуемые документы»: статус каждого = отсутствует / pending / approved / истекает ≤30 дней / просрочен.
- Требуемый набор — конфигурируемый org-список `required_documents` (тип, правило срока: Enrollment = 12 мес от месяца подписи; IEA = до конца FY; medical — при has_health_condition; safepass addendum — подпись активной версии).
- В карточке ребёнка — секция «Documents status»; по центру — сводный экран «у кого чего не хватает» + CSV.

## Фаза 3 — Управление пакетом + агент просроченных (маркетинговая фича)
- **Композер пакета**: директор в карточке ребёнка/семьи выбирает состав — от полного пакета до единичной формы (чекбоксы) → генерируется токен-ссылка (механизм Channel A, 14 дней) на страницу с выбранными формами; профиль автозаполнения §5 работает между формами.
- **Агент просроченных документов**: ежедневный проход по матрице фазы 2 → для каждого ребёнка с истекающими/просроченными документами агент сам собирает точный набор форм + **сопроводительное письмо** (шаблон: какие документы, причина, срок, токен-ссылка) → карточка в разделе **«Неотложные дела» (Urgent)** на дашборде директора → кнопка **Send — один клик**.
- **Эскалационная лестница при игнорировании** (пороги — конфиг, не хардкод):
  - день 0: письмо со ссылкой (один клик директора);
  - день +5, +10: авто-напоминания без участия директора (лог);
  - день +14 (токен истёк) или документ просрочен: карточка в «Неотложных делах» → статус «требует личного контакта», телефон семьи + кнопка «продлить ссылку и отправить снова»; дальше — человек;
  - просроченная IEA: FRP ребёнка автоматически → Paid до новой формы (правило CACFP), директору показывается финансовое последствие.
  - **Момент автопонижения (уточнение Николая, по положению)**: IEA действует до последнего дня месяца её подписи год спустя; понижение F/R → Paid вступает **с 1-го числа следующего месяца**. Пример: подписана 2 июня 2025 → действует по 30 июня 2026 включительно → Paid с 1 июля 2026. Никогда не понижать серединой месяца или в годовщину даты подписи; агент планирует понижение на 1-е число. Предупреждения фазы 2 («истекает ≤30 дней») отсчитываются от конца месяца действия.
- **Автодайджест директору**: понедельничная сводка «истекает на этой неделе у N семей / без ответа M» через send-push; ноль кликов.
- **Канал отправки (решение Николая, 3 июля)**: основной канал всех сношений садик↔родители — **SafePass parent app** (push + обновляемые страницы; доставка логируется). Агент просроченных шлёт push со ссылкой на токен-пакет. **Email — только ручная кнопка «Send by email»** в карточке для семей без приложения (не автоматическая рассылка). Установка SafePass — обязательный шаг онбординга новой семьи (закрывает покрытие). Следствие: бэклог-пункт hardening safepass_sign поднимается в приоритете — SafePass становится магистральным каналом.
- **Правило честной рекламы**: маркетинговые формулировки публикуются только про функции, прошедшие смоук Николая на проде.
- **Маркетинг**: абзац EN+RU в docs/marketing/ («директор одним кликом отправляет семье персональный пакет обновления документов; система сама следит за сроками») — Николай использует в рекламе.
