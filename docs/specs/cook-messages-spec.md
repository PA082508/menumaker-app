## 0. Что оказалось не так в моём же бэклоге

Я записал 403 как «у поварского сервис-аккаунта нет доступа к `internal_messages`».
**Это неверно.** Замер:

```
menumaker.internal_messages — RLS on · политик 0 · грантов anon/authenticated НЕТ · строк 0
```

**Доступа нет ни у кого.** Таблица закрыта для всех ролей (кроме `service_role`, который
обходит RLS). 403 ловит не повар — 403 ловит любой, кто откроет панель: и директор на
`/messages`, и повар на двери. Сообщения в платформе **никогда не работали**.

Это меняет объём: (a) — не «дать повару», а **включить таблицу вообще**.

## 2. (a) Политика чтения — скоуп центра

**Сегодня чтение org-широкое, а не центровое.** [PortalMessagesPanel.tsx:32-33](../../src/pages/messages/PortalMessagesPanel.tsx#L32):

```ts
.eq('org_id', org.id)
.or(`recipient_value.eq.${user?.id},recipient_value.eq.teacher,recipient_value.eq.cook,recipient_value.eq.all`)
```

То есть повар Ridge видит сообщение, адресованное «всем поварам», даже если его писал
директор Pearl про кухню Pearl. Заказ просит скоуп центра — и это надо чинить **в
политике**, а не только в клиентском фильтре: фильтр в браузере не защита.

**Правило (в SQL):** сообщение видно, если
1. `core.is_org_member(org_id)` — базовая изоляция организации, **и**
2. оно адресовано мне: `recipient_value` = мой `auth.uid()`, **либо** моя роль
   (`get_user_role()`), **либо** `'all'`, **и**
3. центр совпадает: `center_id is null` (общеорганизационное) **или** центр в моих
   `core.user_center_access`.

Пункт 3 — это и есть «сообщения своего центра». Он же чинит утечку между центрами,
которая существует прямо сейчас в клиентском фильтре.

**Запись:** отправлять может любой член организации (`with check` на org + себя как
`sender_id`). Кухня отвечает директору — это уже в коде.

⚠️ **`deny_teacher` сюда НЕ вешаем** — RESTRICTIVE ALL-deny убьёт панель у учителя в
день появления роли ([identity-teacher-spec.md](identity-teacher-spec.md) §0.2).

## 3. (c) «Прочитано кухней»

`read_by uuid[]` **уже есть в схеме и уже читается** ([PortalMessagesPanel.tsx:37](../../src/pages/messages/PortalMessagesPanel.tsx#L37)
считает непрочитанные), но **никто в неё не пишет** — счётчик непрочитанных никогда не
обнуляется.

Нужен RPC, а не `update` из клиента: иначе политика записи должна разрешить
`update` на чужое сообщение, что открывает правку `body`.

```sql
menumaker.mark_message_read(p_message uuid) -- SECURITY DEFINER
-- append auth.uid() to read_by, если его там нет; проверяет, что сообщение видно
-- вызывающему по тем же правилам, что и политика чтения. Идемпотентно.
```

⚠️ **`read_by` — это «кто из людей открыл», а не «кухня приняла».** На общем
сервис-аккаунте двери ([[menumaker-no-teacher-role]]) `auth.uid()` — это **дверь**, а не
человек: «прочитано кухней Ridge» будет значить «кто-то у этого iPad открыл панель».
Для сообщения это, вероятно, достаточно. **Если нужна подпись человека — это PIN**
(та же развилка, что в attendance, [identity-teacher-spec.md](identity-teacher-spec.md) §4). Вопрос §5.

## 4. (d) Адресация «кухня центра» — механика НЕ умеет

[MessagesPage.tsx:23-26](../../src/pages/messages/MessagesPage.tsx#L23):

```ts
{ value: 'teacher', label: '👩‍🏫 All Teachers' },
{ value: 'cook',    label: '👨‍🍳 All Cooks' },
{ value: 'director',label: '📊 All Directors' },
{ value: 'all',     label: '📢 Everyone' },
```

Получатели — **org-широкие роли**. «Все повара» ≠ «кухня Ridge». Заказ это предвидел:
«если механика получателей не умеет — в спеку». Не умеет.

**Но изобретать `recipient_type` не нужно.** На сообщении **уже есть `center_id`**, и он
уже проставляется при отправке. Значит «кухня центра» = `recipient_value='cook'` +
`center_id=<центр>`, а «все повара организации» = `recipient_value='cook'` +
`center_id=null`. Одна колонка, ноль новых понятий.

Нужно:
1. **UI директора:** к выбору роли — выбор центра (или «вся организация»). Сегодня
   `MessagesPage` `center_id` при отправке **не проставляет вовсе** — только
   `PortalMessagesPanel` проставляет.
2. **Клиентский фильтр** привести в соответствие с политикой (центр + роль), чтобы UI
   не показывал меньше/больше, чем разрешает БД.
3. Ярлык: «👨‍🍳 Кухня — Wickliffe» вместо «Все повара».
