# RENEWAL-КОНТУР — спека

**Статус:** на утверждение Николаю. Кода нет. Дата: 2026-07-16.
**Дедлайн кампании:** 31.07.2026.
**Правило:** реестрово, без хардкода «кампании июля». Кампания — это **штатный режим**, а не спецслучай.

---

## 0. Заказ в одну фразу

При кампании обновления **ручной осмотр только там, где нужна подпись директора**.
Всё остальное — авто в БД, со знанием админа.

Из этого следует инверсия сегодняшней логики. Сейчас Inbox — это очередь **всего**, что
пришло (`status='pending'`, [EnrollmentInboxPage.tsx:143](../../src/pages/enrollment/EnrollmentInboxPage.tsx#L143)), и человек глазами
перебирает каждую строку. При 150 продлениях эта очередь перестаёт быть сигналом.
Контур должен оставить в очереди **только то, что требует человека**, а остальное
провести в БД и показать как факт, а не как задачу.

---

## 1. Главный дом контура = тетрадь ENROLLMENT

Renewal Tracker, очередь «Awaiting director signature», расхождения и «кому ещё послать»
живут **внутри Enrollment**. Не на аватарках, не отдельной страницей.

### 1.1. Красная цифра = только actionable

Сегодня цифра считается в двух местах ad-hoc и означает «сколько pending»
([CenterRosterPage.tsx:239](../../src/pages/children/CenterRosterPage.tsx#L239), [StaffPage.tsx:53](../../src/pages/staff/StaffPage.tsx#L53)).

**Новое определение — цифра считает только то, что ждёт человека:**

| Входит в цифру | Не входит |
|---|---|
| ждёт подписи директора (`requires_countersign=director`) | авто-филенное (`received`) |
| расхождение при матчинге / не сматчено | успешно продлённое |
| new enrollment (не renewal) | — |
| *(опционально, флагом)* «не отправлено» | — |

Если авто-филенное попадёт в цифру, сигнал снова станет «150» и умрёт. Это главное
требование к контуру, а не деталь.

Авто-филенное **видно** — внутри трекера, со статусом `received`. Видно ≠ actionable.

**Инженерная заметка:** цифру надо считать один раз и в одном месте. Сегодня она
дублируется двумя запросами `count:'exact', head:true` без refetch после approve.
Кандидат — блок в `refresh_action_items`, где уже есть severity/dismiss/snooze
([ActionItemsWidget.tsx:49](../../src/components/dashboard/ActionItemsWidget.tsx#L49)), но enrollment там сейчас не представлен.
Решение — при утверждении.

## 2. Состав контура

### (a) Реестровый признак `auto_file`

Новое поле в записи реестра `enroll-registry.json` (schema 2).

```jsonc
"iea": {
  "auto_file": true,          // сабмит, сматченный с ребёнком ростера + валидный → авто-файл
  "expiryMonths": 12,         // уже есть — цикл продления
  "signer": "parent",         // уже есть
  ...
}
```

**Правило авто-файла — все три условия одновременно:**

1. это **renewal** (сабмит однозначно сматчен с ребёнком активного ростера — см. §2e), **и**
2. валидация чистая (🟢 по `validateSubmission`, без errors), **и**
3. форма несёт `auto_file: true` **и** не требует контрподписи (§2b).

→ пишем в БД, статус **`received`**, **без ручного Approve**.

**Всё остальное — в ручную очередь как сейчас:**
- new enrollment (нет матча с ростером),
- любое расхождение (см. §2e),
- 🟡/🔴 валидация,
- формы без `auto_file`.

> **Что НЕ авто-филится никогда, независимо от флага:** определение F/R/P
> (`approveIea` → `roster.frp` + `income_eligibility`). Это claim-доказательство и оно
> под claim-мостом до 1 октября. IEA может авто-филиться как *документ получен*, но
> **определение F/R/P остаётся ручным** до отдельного решения. Это граница, которую
> контур не переходит.

**Миграция статусов.** `status` в БД ограничен CHECK-ом
`('pending','approved','rejected')` ([20260703_enrollment_submissions.sql:32](../../supabase/migrations/20260703_enrollment_submissions.sql#L32)).
`received` требует миграции + правки `.eq('status','pending')` в Inbox и в обеих
badge-запросах. Миграция — prepare+go, не автоматом.

### (b) `requires_countersign` — всегда в рабочую очередь

Нового поля в реестре **нет** (есть `signer: 'parent' | 'physician'`).

```jsonc
"requires_countersign": "director"   // новое; null | "director" | "staff"
```

Форма с `requires_countersign: "director"` **никогда** не авто-филится, даже при
`auto_file: true` и чистой валидации. Она идёт в очередь с фильтром
**«Awaiting director signature» — включённым по умолчанию**.

Известные кандидаты: Transition into program (центр-сторона = директор на Inbox review),
Release Authorization v2. Точный список — при утверждении.

Сегодня в Inbox **нет ни одной вкладки/фильтра** — есть только поиск по имени и
URL-параметр `?from=staff|children`. Фильтры-вкладки строятся с нуля.

### (c) Renewal Tracker (MVP)

Внутри Enrollment. По семьям/детям, колонки = стадии:

| Стадия | Источник истины |
|---|---|
| **Отправлено** (prefill выдан) | нужна новая таблица — сегодня факт отправки нигде не фиксируется |
| **Заполнено** | `enrollment_submissions.created_at` |
| **В БД** | `status='received'` (авто) или `'approved'` (ручное) |
| **Ждёт подписи** | `requires_countersign` + подписи нет |
| **Готово** | подписано / продление закрыто |
| **Кому ещё послать** | ростер минус «отправлено» — колонка-остаток |

Последняя колонка — **самая ценная для админа** и единственная, которую нельзя
получить из существующих таблиц: нужен реестр «кому что выдано». Без него кампания
идёт вслепую.

**Это же — ответ на «кому ещё послать»**: не отдельный экран, а колонка трекера.

### (d) Бейдж на аватаре — вторичный слой

⏳ = отправлено/ждём · ✓ = обновлено. Виден при открытии класса.
**Повседневный сигнал остаётся за красной цифрой Enrollment** — бейдж её не заменяет.

`Avatar.tsx` не имеет слота под бейдж, но ростер уже кладёт оверлей снаружи —
`position:relative` обёртка + абсолютный кружок ([CenterRosterPage.tsx:698](../../src/pages/children/CenterRosterPage.tsx#L698)).
**Важно: правый нижний угол занят точкой присутствия** — renewal-бейдж идёт в правый
верхний. Либо расширить `Avatar` необязательным `badge?: ReactNode` (безопасно: все
существующие вызовы передают только `name`/`size`/`path`/`fontSize`).

### (e) Матчинг renewal → ребёнок

Логика **уже есть** и переиспользуется целиком — `matchRoster`
([enrollmentApprove.ts:145](../../src/lib/enrollmentApprove.ts#L145)):

- точное совпадение нормализованного имени (в любом порядке или по `child_name`), **либо**
- мягкое совпадение (ограниченный Левенштейн) **только если DOB подтверждает**;
- **противоречащий DOB исключает всегда**;
- `loadCenterRoster` намеренно включает неактивных — вернувшийся ребёнок реактивируется, а не дублируется.

**Критерий авто-файла — строже, чем критерий показа:**

| Результат матчинга | Действие |
|---|---|
| ровно 1 кандидат, точное имя + DOB совпал | **renewal → авто-файл** (при прочих условиях §2a) |
| ровно 1 кандидат, мягкое имя + DOB подтвердил | **renewal → авто-файл** |
| ровно 1 кандидат, имя точное, DOB отсутствует | **в ручную очередь** — не гадаем |
| 2+ кандидата | **в ручную очередь**, показать всех кандидатов |
| 0 кандидатов | **new enrollment** → ручная очередь |
| DOB противоречит | **в ручную очередь** (никогда не матч) |

**Правило: при любой неоднозначности — в ручную очередь. Не гадать.**
Неверный авто-матч тихо пишет документ в чужую карточку — это хуже, чем лишняя
строка в очереди.

**Издание/фискальный год** берётся из **строки издания формы**, не из арифметики по
датам (`parseIeaFiscalYear`, [enrollmentApprove.ts:93](../../src/lib/enrollmentApprove.ts#L93)). Продлить = указать на новое
издание (`current` в реестре), и FY выпадает из токена сам. Это же согласуется с
queued-спекой «period-effective references».

## 3. Коммерческое применение

Этот же контур — **zero-migration onboarding**.

Когда клиент переходит на платформу, у него нет истории в нашей БД: нужно массово
собрать актуальные документы по всему ростеру. Это ровно кампания продления, только
стартовое состояние — «не отправлено» у всех.

Отсюда проектные требования:

1. **Никакого хардкода «июль 2026»**, никаких `if (campaign === 'renewal_july')`.
   Кампания — сущность: набор форм (из реестра) × набор детей (ростер/фильтр) × срок.
2. **Массовое обновление документов = штатный режим**, а не спецслучай. Один и тот же
   трекер обслуживает и продление, и онбординг нового клиента.
3. Состав кампании **выводится из реестра** (`expiryMonths`, `perInstance`, `condition`,
   `packets`), а не перечисляется руками.
4. Депersonalization уже требуется (center-lock + org-branding из настроек, `org` в
   реестре — Stage-2 swap point) — контур обязан её уважать.

**Коммерческий смысл:** «переход к нам не требует миграции данных — вы запускаете одну
кампанию, и через неделю ваш ростер в системе с актуальными подписанными документами».
Это снимает главное возражение при продаже. Контур строится под это с первого дня, а
не переделывается потом.

## 4. Оценка

| Блок | Объём | Риск | Зависимости |
|---|---|---|---|
| (a) `auto_file` в реестре + правило | S | низкий | обе копии реестра синхронно |
| (a) миграция `status: received` | S | **средний** — CHECK + 3 места чтения | prepare+go |
| (b) `requires_countersign` + фильтр-вкладки | M | низкий | вкладок в Inbox нет вообще — с нуля |
| (c) таблица «выдано/отправлено» | M | средний — новая сущность | миграция, prepare+go |
| (c) Renewal Tracker UI | **L** | средний | зависит от (a)+(c) |
| (d) бейдж на аватаре | S | низкий | после (c) |
| (e) матчинг — переиспользование | S | низкий | `matchRoster` готов |
| Красная цифра: свести в одно место | M | средний — трогает 2 живых экрана | решение по `action_items` |

**Критический путь к 31.07:** (c) таблица «отправлено» → (a) авто-файл → (c) трекер.
Без таблицы отправки нет ни «кому ещё послать», ни стадии «отправлено» — то есть нет
трекера. **Это первое, что надо утвердить.**

**Предлагаемая нарезка:**
- **Волна 1 (разблокирует кампанию):** таблица отправки + `auto_file` + `received` + очередь «Awaiting director signature». Даёт: авто-файл работает, очередь = только actionable.
- **Волна 2:** Renewal Tracker UI + «кому ещё послать».
- **Волна 3:** бейдж на аватаре, консолидация красной цифры.
