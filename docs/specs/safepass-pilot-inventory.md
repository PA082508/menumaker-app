## 0. ⚠️ Противоречие в постановке — решить до кода

| Заказ | Факт |
|---|---|
| «Киоск-WIP в пилот **НЕ входит** — подтвердить, что пилотный поток его не требует» | **Требует.** Подтвердить обратное невозможно |
| «SafePassTeacherPage перевести на `safepass_confirm_handoff` (PIN, `teacher_id = staff.id`)» | Эта RPC **token-gated** |

`safepass_confirm_handoff` первым делом:
```sql
select * into v_dev from menumaker.safepass_devices
 where token_hash = encode(digest(p_token,'sha256'),'hex') and is_active and revoked_at is null;
if not found then raise exception 'device not registered'; end if;
```
Токен выдаёт только `safepass_register_device`; хранит его `safepassDevice.ts`
(`localStorage['sp_kiosk_token']`). **PIN тоже приходит оттуда**: `pin_hash = sha256(center_id:pin)`,
а `center_id` берётся из устройства.

> **Перевести приём на правильную процедуру = включить киоск-механику.** Это одно и то же
> действие, названное двумя способами. Третьего пути к «`teacher_id = staff.id`» нет:
> без устройства RPC не вызвать, без RPC подписи нет.

**Развилка (решение Николая):**
- **(A)** Киоск входит в пилот → ревью WIP (`safepassDevice.ts`, `shared/PinPad.tsx`,
  `shared/kioskUI.tsx` — все untracked) + добить регистрацию и PIN. Честно, но это не «правка до понедельника».
- **(B)** Киоск не входит → приём остаётся прямым `.update()` без PIN, `teacher_id` = сервис-аккаунт.
  Тогда цель «первый ребёнок принят правильной процедурой» **не достигается**, и пилот пишет
  записи в старом формате — ровно то, чего заказ хотел избежать.
- **(C)** Пилот сдвигается.

### 1.4. Путь родителя — **НУЖЕН КОД (два блокера) + НЕТ ДАННЫХ**

**Блокер 1 — заявка физически не может попасть в очередь Red Room.**
[SafePassParentPage.tsx:87](../../src/pages/safepass/SafePassParentPage.tsx#L87) захардкожено:
```ts
classroom_id:'', classroom_name:'Green Room', center_id:p.center_id||''
```
при вставке (`:148`) `classroom_id: selectedChild.classroom_id || ORG_ID` → в сессию уходит
**ORG_ID**. Учительский экран фильтрует `.eq('classroom_id', classId)` и realtime по
`classroom_id=eq.<uuid комнаты>`. **Org-UUID никогда не равен UUID комнаты → заявка не появится
никогда.** Родитель будет вечно смотреть на «ожидание». Плюс родителю Red Room экран скажет
«Green Room».

**Блокер 2 — OTP не существует.** [:61-65](../../src/pages/safepass/SafePassParentPage.tsx#L61):

консоль**, и сверяется (`:76`) с ним же. **SMS нет нигде** — ни Twilio, ни edge-функции
(`supabase/functions/` = `cacfp-meal-check`, `enrollment-scan-ocr`, `provision-access`, `send-push`).
Плюс бэкдор `TEST_PHONES` (`:32`) на публичном маршруте.

## 2.6. Правка инвентарного запроса (на будущее) — 2026-07-18

Инвентарь фильтровал Red-персонал как `is_active AND classroom_id=<Red>` и вернул «пусто»,
хотя активные учителя Red есть — у них `classroom_id=NULL` (раздвоение поколений: активная
строка 25.06 без комнаты + неактивная 21.06 с комнатой; см. readiness-27 §2.2).
**`active + classroom` даёт ЛОЖНОЕ «пусто» при NULL-комнате.** Инвентарный запрос по `staff`
должен выводить **отдельной секцией строки `is_active AND classroom_id IS NULL`** («active без
комнаты») — тогда раздвоение видно сразу, а не находится вручную. Плюс сигнал дублей:
несколько строк на одно имя в центре = кандидат на дедуп (не только Red — `Colleen Minadeo`
тоже раздвоена). См. дедуп-очередь «staff-поколения» ([[menumaker-backlog]]).
