### 2.3. Путь родителя — три блокера

**(1) `classroom_id` — заявка не дойдёт до комнаты.**
[`SapePassParentPage.tsx:87`](../../src/pages/safepass/SafePassParentPage.tsx#L87) захардкожен
`classroom_id:''`, `classroom_name:'Green Room'`; при вставке уезжает `ORG_ID`. Учительский
экран фильтрует по UUID комнаты → **совпадения не будет никогда**. Плюс родителю Red
экран скажет «Green Room».
**Цена:** мал — взять `classroom_id` из `roster` по `child_id` доверенного лица.

**(2) OTP — его не существует.**
Код генерируется **в браузере родителя**, кладётся в его же `sessionStorage`, печатается
в консоль; плюс `TEST_PHONES`-бэкдор на публичном маршруте. SMS нет нигде.

**✅ Вариант `safepass_temp_codes` проверен — форма таблицы подходит:**
```
id, org_id, center_id, child_id, child_name, code, person_name,
issued_by, issued_at, expires_at, used_at, used_session, is_active
```
Политики: `org_member_all` (authenticated) + `service_role_all`. **anon-политики НЕТ** →
висячий `anon:SELECT` глушится RLS. Строк 0, **RPC не касается её ни один**.

**Что это значит:** таблица уже описывает ровно модель «директор выдал код офлайн»
(`issued_by`, `expires_at`, `used_at`, `used_session`). Но **родитель ходит как anon**, а
anon читать её не может и **не должен** — иначе повторим дыру `safepass_sms_otp`, которую
вчера закрыли.

**Нужно:**
- RPC `safepass_redeem_temp_code(p_code, p_phone)` — SECURITY DEFINER, `grant execute to anon`;
  сверяет код, отмечает `used_at`/`used_session`, возвращает ребёнка. **Anon никогда не SELECT-ит коды.**
- `revoke select on safepass_temp_codes from anon` — висячий грант убрать (defence-in-depth).
- UI у директора: выдать код на ребёнка, показать/распечатать.
- `SafePassParentPage`: вход по коду вместо фейкового OTP; **`TEST_PHONES` удалить**.

**Цена:** средняя. Это меньше, чем поднимать SMS-провайдера (ключи, деньги, номер), и
**не требует внешних зависимостей к 27-му**.

**(3) Фейковый «Test Child».**

# Поверхность Devices — заказ 18.07 (экран + карточка в Hub одним заходом)

**DoD, зафиксированный Николаем:** «фича = код + инструкция». Инструкции
Devices/offline поэтому **не пишутся вперёд экрана** — они пишутся сверкой с
живым UI, иначе получится документ про воображаемые кнопки. Ниже — заказ на то,
из чего инструкция потом вырастет.

## 1. Экран (директор)

Регистрация iPad уже есть в коде клиента (`safepassDevice.registerDevice` →
`menumaker.safepass_register_device`, гейт: член org + роль
director/office_manager/admin). Не хватает **поверхности**, а не механики.

Экран обязан уметь ровно три вещи:

1. **Зарегистрировать планшет** — выбрать центр + комнату, дать метку,
   получить токен. Токен показывается **один раз**; на экране это должно быть
   написано словами, а не подразумеваться.
2. **Показать список устройств** — комната, метка, когда зарегистрировано,
   активно/отозвано. Без списка отзыв вслепую.
3. **Отозвать доверие** — `revoked_at` + `is_active=false`. **Обязательная
   функция, не опция:** планшет теряется, и это единственный способ закрыть
   дверь. Отзыв не удаляет строку — история приёмов остаётся привязанной.

Перерегистрация после отзыва = новая строка с новым токеном. Восстановления
потерянного токена не существует и не должно появиться.

## 2. Карточка в Doc Hub — тем же заходом

Не отдельной задачей. Одна карточка директора, три строки:

- **как зарегистрировать iPad** (со ссылкой на экран, а не пересказом);
- **как отозвать** — и явно: делать это сразу, а не «когда дойдут руки»;
- **как ведёт себя приём без сети** — одной строкой, без обещаний: *сегодня
  приём без сети не проходит; связь в комнате обязательна.*

Про офлайн в карточке директора — именно одна строка. Развёрнуто — в
`teacher-portal-order.md`, секция «Офлайн», где разведены офлайн-признак и
полный офлайн (Stage 2: кэш ростера + очередь досылки).

## 3. Чем это НЕ является

Регистрация **классного iPad** — ручной белый список директора, и это здесь
правильная модель: планшет принадлежит центру, его заводит и отзывает человек.

**Родительское устройство — противоположная модель** (канон Николая 18.07,
`DECISIONS.md` §6): само-регистрация по персональной ссылке, телефон =
доверенный носитель, отзыв доверия обязателен, смена телефона = повторная
само-регистрация той же ссылкой.

## Осей на самом деле ТРИ, а не две

Уточнение после разбора Concept v1.1 + BYOD-пакета (18.07). Смешивать их нельзя
— у каждой свой владелец доверия и свой жизненный цикл:

| ось | кто заводит | отзыв | источник |
|---|---|---|---|
| **классный iPad** | администрация, белый список | `revoked_at`, вручную | Devices-экран |
| **staff BYOD** — личный телефон сотрудника | администрация по подписанному соглашению | **≤ 24 ч при увольнении** | HR-BYOD-001 |
| **родительское устройство** | **сам родитель** по персональной ссылке | по заявлению + при утере | Parent Letter |

**Staff BYOD — то, чего в системе нет вообще.** По HR-BYOD-001 требуется:

- реестр устройств сотрудников: make / model / номер телефона **из соглашения**,
  а не из головы;
- стипендия **$20/мес** → это стык с Payroll, а не косметика;
- **офбординг ≤ 24 часов при увольнении — обязательная функция**, не опция:
  уволенный сотрудник с живым доступом к детским данным есть инцидент, а не
  недоделка;
- увязка со Staff-модулем: устройство висит на строке сотрудника, гаснет вместе
  с ней.

Ничего из этого не построено.

**Решение Николая 18.07: это НЕ отдельная спека — это секция Staff-модуля**, с
интерфейсным флагом наружу для SafePass. Раскладка по владельцам:

| часть | куда ложится |
|---|---|
| реестр устройств (make / model / phone из соглашения) | **свойство сотрудника** — строка в Staff, как телефон или должность |
| стипендия $20/мес | **Payroll** — не косметика, а регулярная выплата |
| офбординг ≤ 24 ч | **termination-процесс**, а не ручная задача: увольнение обязано гасить устройство само |
| «это устройство допущено к SafePass» | **интерфейсный флаг** — единственное, что SafePass читает |

Смысл раскладки: SafePass не должен знать про стипендии и увольнения, а Staff не
должен знать про токены. Между ними — один флаг.

**Терминологическая ловушка:** фраза «devices registered and authorized by
administration» из старых текстов — это про **BYOD сотрудников**, а не про
родителей. Родительский абзац формулируется осью само-регистрации. Одна фраза,
две оси — ровно тот случай, когда копирование формулировки переносит чужую
модель доверия.
