# Ратификация Николая 18.07 — ответы по данным

## Двойное представление «учитель ↔ класс» — не противоречие, а два поля

Экран класса Николая показывает `Teachers: Maureen, Carolyn`, а мой замер говорил
«к Red не привязан никто». Оба верны — это **два независимых представления одной
связи**, и они разошлись:

| | поле | тип | кто читает |
|---|---|---|---|
| **для глаз** | `staff.class_primary` | TEXT, имя комнаты | ВСЕ экраны с «Teachers» |
| **для механики** | `staff.classroom_id` | FK uuid | **никто** |

Измерено: Carolyn (`84401340`) и Maureen (`d98ebb4e`) — обе `class_primary='Red'`,
`classroom_id=NULL`. Экран матчит нормализованным ИМЕНЕМ комнаты
([ChildrenPage.tsx:251-258](../../src/pages/children/ChildrenPage.tsx#L251-L258),
[CenterRosterPage.tsx:568-570](../../src/pages/children/CenterRosterPage.tsx#L568-L570)),
поэтому обе видны. SafePass/handoff же ходит по FK — поэтому механика их не видит.

**FK не читает ни одна поверхность в коде.** Это и чинит `20260718d` (Red сейчас)
+ правка StaffSettings UI (остальные).

⚠️ Третья поверхность строже: печатная посещаемость
([AttendanceBlankReport.tsx:103-105](../../src/pages/reports/AttendanceBlankReport.tsx#L103-L105))
матчит `class_primary` СТРОГИМ равенством, без нормализации — молча пустеет при
любом расхождении регистра/пробелов. Спека должна свести все три к одному пути.

### ⚠️ Ловушка миграции text → FK

Наполненность текстового поля выше, чем я показывал по FK:

| Центр | живых | `class_primary` заполнен | FK заполнен | **текст НЕ совпал ни с одной комнатой** |

# Офлайн — что есть сейчас, и что офлайном ещё не является (18.07)

Секция честная намеренно: слово «офлайн» уже звучит в разговорах о портале, и
если не развести два разных смысла, спека унаследует обещание, которого код не
даёт. **Офлайн — свойство ПРИЁМА, а не настройка**: нигде нет и не будет
переключателя «офлайн-режим». Есть признак, который сервер проставляет сам.

## Что офлайн есть СЕЙЧАС

**1. Офлайн-признак подтверждения.** Семантика — по коду
`menumaker.safepass_confirm_handoff`, не по памяти:

```sql
v_offline := p_occurred_at < now() - interval '5 seconds';
update menumaker.safepass_sessions
   set offline_created   = coalesce(offline_created, v_offline),
       offline_synced_at = case when v_offline then now() else offline_synced_at end
```

Читается так:

- `p_occurred_at` по умолчанию `now()`. Обычное подтверждение при живой сети
  приходит с сервера же → разница ≈ 0 → `v_offline = false`.
- Клиент передаёт `p_occurred_at` **только** при досылке отложенного события
  (`safepassDevice.confirmHandoff(..., occurredAt?)`). Текущая проводка
  teacher-страницы его не передаёт — все её подтверждения честно онлайновые.
- Порог **5 секунд** — это «сетевая задержка», а не «связь была». Отсюда прямое
  следствие для спеки: **признак меряет расхождение часов планшета с сервером.**
  Планшет с уехавшими часами пометит онлайновый приём офлайновым и наоборот.
  Пока досылки нет, цена ошибки нулевая; с приходом очереди (Stage 2) признак
  надо будет либо привязать к факту постановки в очередь, либо честно назвать
  эвристикой в UI. **Не строить на нём отчётность, пока это не решено.**
- `coalesce(offline_created, v_offline)` — уже проставленный признак повторной
  досылкой не перетирается.
- Приём идемпотентен: `status='confirmed'` → ранний возврат `already:true`,
  без повторной записи. Двойное нажатие безопасно.

**2. Троттлинг PinPad переживает перезагрузку.** `sp_pin_lock` в localStorage:
4 неверных PIN → 45 с блокировки, лок сохраняется через reload и работает без
сети. Попытку сжигает **только** `InvalidPinError`; сетевая ошибка попытку не
жжёт. Блокировка на устройство — правильный масштаб для общего планшета.

## Что офлайном ещё НЕ является — Stage 2

Ни того, ни другого сегодня нет:

- **кэш ростера** — без сети список детей не отрисуется вообще;
- **очередь отметок с досылкой** — подтверждение без сети просто не уйдёт;
  RPC-вызов упадёт, и запись не появится.

**Для attendance это критично**, потому что строка посещаемости — производная
handoff-события: нет события — нет строки, и восстановить её потом нечем. То
есть отсутствие очереди — это не деградация удобства, а пропуск в учёте.
